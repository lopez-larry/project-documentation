## Document Types

### 1. Project Charter

* **Purpose:** Authorizes the project, defines objectives, scope, and key risks.
* **How to Write:**

  * Use short bullets, not long paragraphs.
  * Required: Purpose, Objectives, Scope (In/Out), High-Level Risks, Approvals.
* **How to Read:**

  * Executives/Sponsors â†’ Focus on Purpose, Objectives, Risks.
  * PM/BA â†’ Read full doc.
  * Developers/QA â†’ Skim Objectives and Scope.

### 2. Software Requirements Specification (SRS)

* **Purpose:** Defines **what** will be built â€” functional + non-functional requirements.
* **How to Write:**

  * Use IDs (FR-01, NFR-02).
  * Functional: testable behaviors.
  * Non-Functional: performance, security, compliance.
  * Keep descriptions short and testable.
* **How to Read:**

  * BA/PM â†’ Review Scope, Requirements, Use Cases.
  * Developers/QA â†’ Read all requirements.
  * Executives â†’ Skim Objectives and Summary.

### **2.5. Architecture Contract Specification (ACS)**

* **Purpose:** Defines the **system boundary contract** â€” what the system guarantees externally (schemas, APIs, error rules, security, service-level objectives).

  * Bridges SRS (*what must be built*) and SDD (*how it will be built*).
  * Establishes measurable, stable guarantees the architecture must deliver.

* **How to Write:**

  * Bullet each contract element:

    * External Data Schemas (NOT database schema)
    * API Operations (inputs/outputs)
    * Error Handling Rules
    * Security & Access Policies
    * Quality Objectives (performance, availability, reliability)
    * Versioning & Compatibility Rules
    * Acceptance Criteria (Definition of Done)
  * Keep content consumer-focused and implementation-agnostic.
  * Include appendices for contract creation meetings, workflow, and governance.

* **How to Read:**

  * Architects â†’ Read entire document.
  * Developers â†’ Read API, Error, Security, and Acceptance Criteria.
  * QA â†’ Read Error Contract, SLOs, and Acceptance Criteria.
  * PM/Product Owner â†’ Review high-level guarantees and constraints.

### 4. Test Plan

* **Purpose:** Defines how requirements will be validated.
* **How to Write:**
